# Login Extension Test Mode Design

## 목표

- 로그인 연장 동작을 1시간 기다리지 않고 실제 staging 화면에서 확인함.
- 클릭 실행 여부와 버튼 탐색 실패 이유를 출력 화면에서 바로 확인함.
- 기존 selector 안전 조건과 계정정보 비저장 원칙을 유지함.

## 선택한 방식

- 첫 실행과 반복 주기를 테스트용 1분으로 변경함.
- 클릭 스크립트가 실행될 때 웹뷰 우상단에 결과 알림을 5초간 표시함.
- 동일 결과를 개발자 콘솔에도 기록함.
- 관리 화면에 별도 이력 저장이나 IPC 통신은 추가하지 않음.

## 결과 구분

- 후보 1개: 버튼의 `click()`을 호출하고 `로그인 연장 클릭 실행`을 표시함.
- 후보 0개: 클릭하지 않고 `로그인 연장 대상 버튼 없음`을 표시함.
- 후보 2개 이상: 클릭하지 않고 `로그인 연장 대상 버튼 여러 개`를 표시함.
- 알림에는 실행 시각을 함께 표시함.

## Selector 안전 조건

- `<button>` 요소여야 함.
- `stamp`와 `stamp--normal` class를 모두 가져야 함.
- 전달받은 em dash 표기 `stamp—normal`도 허용함.
- 정리한 텍스트가 `^\d{2}:\d{2}:\d{2}$`와 일치해야 함.
- 비활성, `aria-disabled`, 숨김, 크기 0 요소는 제외함.
- 후보가 정확히 하나일 때만 클릭함.

## 알림 처리

- 고정된 DOM id를 사용해 웹뷰마다 알림을 하나만 유지함.
- 새 실행 결과가 오면 기존 알림 내용을 교체하고 제거 타이머를 다시 시작함.
- 알림은 `pointer-events: none`으로 페이지 조작을 방해하지 않음.
- 알림 생성 실패가 버튼 클릭 자체를 막지 않도록 클릭과 기록을 독립적으로 처리함.

## 대안과 판단

- 관리 화면 이력: 영구 기록에는 유리하지만 원격 웹뷰 결과를 Rust로 전달하는 IPC가 추가되어 테스트 목적보다 범위가 큼.
- 콘솔 기록만 사용: 구현은 가장 작지만 출력 중 사용자가 바로 확인하기 어려움.
- 웹뷰 알림과 콘솔 기록을 함께 사용하는 방식을 선택함.

## 검증

- Rust 상수가 60초인지 테스트함.
- 단일 후보에서 클릭과 성공 알림이 함께 발생하는지 DOM 테스트함.
- 후보 없음과 복수 후보에서 클릭 없이 각각의 실패 알림이 표시되는지 테스트함.
- 기존 숨김·비활성·class 조건 테스트를 유지함.
- 전체 TypeScript·Rust 테스트, 타입 검사, Clippy, 빌드를 실행함.
- 최신 Tauri 앱을 재실행해 1분 테스트가 가능한 상태로 둠.

## 복구

- 실제 검증 완료 후 `LOGIN_EXTENSION_INTERVAL`을 1시간으로 되돌리고 테스트용 웹뷰 알림 제거를 권장함.
